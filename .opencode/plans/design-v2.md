# 等保合规智能平台产品设计方案（V2.0）

---

## 一、产品概述

### 1.1 产品定位

基于 AI Agent + MCP 架构的实战化等保合规自检平台，覆盖等保二级/三级，支持云上与云下部署。

### 1.2 核心痛点与解决思路

| 痛点 | 传统方式 | 我们的解决思路 |
|------|----------|----------------|
| 纸上合规 | 配置打勾 = 合规，实际防线无效 | 白盒配置 + 黑盒渗透交叉验证，打出"纸上合规"原形 |
| 效率低下 | 人工逐项核查，周期长、成本高 | AI 调度工具自动执行，技术域核查效率提升 10x |
| 管理域薄弱 | 制度文档人工审查，主观性强 | AI 语义分析制度文档，自动匹配条款覆盖度 |
| 一次性合规 | 拿到报告就束之高阁，合规状态持续退化 | 持续监控 + 变更触发重评 + 趋势看板 |
| 整改断层 | 发现问题后无闭环，整改跟踪靠人工 | AI 生成整改方案 -> 工单派发 -> 一键复测 |

### 1.3 合规等级覆盖

| 等级 | 覆盖策略 | 说明 |
|------|----------|------|
| 二级 | 轻量版/高级版均可 | 面向中小企业，自检为主，覆盖基本要求 |
| 三级 | 轻量版/高级版均可 | 面向关键业务系统，含深度验证，市场需求最大 |
| 四级及以上 | 不做自动化 | 涉及国家关键基础设施，需人工专家评审 |

用户在项目创建时自由选择目标等级（二级/三级），系统自动加载对应条款集与判定规则。

### 1.4 产品愿景叙事

#### 1.4.1 核心信念

**配置合规 ≠ 防线有效。真正的合规是能防住攻击。**

传统等保合规停留在"填表打勾"阶段，测评通过不代表安全。我们的信念是：合规不应该是一次性的 paperwork，而应该是持续的、可验证的、能真正防住攻击的实战能力。

#### 1.4.2 一句话定位

**"让等保合规从'纸上谈兵'变成'实战验证'"**

#### 1.4.3 终局愿景

成为等保合规领域的"事实标准"：
- 测评机构用我们的预检报告作为参考基线
- 云厂商将我们的合规引擎嵌入等保合规套餐
- 企业客户将我们作为持续合规的操作系统

#### 1.4.4 3-5 年演进路径

| 阶段 | 定位 | 关键能力 |
|------|------|----------|
| 当前 | 等保自检工具 | AI Agent + MCP 自动化检查 |
| 1-2 年 | 合规操作系统 | 持续监控 + 整改闭环 + 多标准支持（ISO27001、网络安全法） |
| 3-5 年 | 合规生态平台 | 接入测评机构、云厂商、企业客户，形成数据飞轮 |

### 1.5 核心价值主张（分角色）

| 角色 | 核心诉求 | 产品价值 | 量化收益 |
|------|----------|----------|----------|

| 云厂商 | 增值服务能力 | 捆绑销售、提高客户粘性 | 等保合规套餐客单价提升 |

### 1.6 Aha Moment 定义

| 用户类型 | Aha Moment | 触发条件 | 心理反应 |
|----------|------------|----------|----------|
| 技术型用户（轻量版） | 第一次看到"纸上合规"被击穿 | 上传截图后，AI 识别出密码策略不符合要求 | "原来配置符合不代表安全" |
| 技术型用户（高级版） | 第一次看到攻击路径图 | 云 API 读取配置显示"符合"，但渗透验证成功绕过 | "原来我的防线可以被绕过" |
| **低技术能力用户（轻量版/高级版均可）** | **第一次用自然语言完成安全检查** | **输入"帮我看看服务器安不安全"，AI 自动完成扫描、截图审查、生成报告，全程无需手动配置** | **"原来不需要请安全专家，我自己就能搞定等保"** |

#### 低技术能力用户 Aha Moment 触发路径

```
小企业老板/兼职IT
    |
    v
注册 -> 输入 1 个公网 IP -> 输入"帮我看看服务器安不安全"
    |
    v
AI 自动执行:
  1. nmap 扫描端口（用户看着工具卡片动，不需要懂 nmap）
  2. nuclei 扫漏洞（用户不需要懂漏洞原理）
  3. 截图引导："请截一张服务器登录界面的图"（用户只需要截图）
  4. AI 自动识别截图中的风险
    |
    v
3 分钟后，AI 输出:
  "您的服务器存在 3 个高危风险:
   1. SSH 端口对全网开放，建议限制访问 IP
   2. 发现 1 个已知漏洞，需要升级补丁
   3. 密码策略过弱，建议设置复杂密码
   [查看完整报告] [一键生成整改方案]"
    |
    v
Aha Moment: "我什么都不懂，但 3 分钟就拿到了安全体检报告"
    |
    v
引导付费: "免费版可扫描 1 个 IP，升级后可扫描全部资产 + 持续监控"
```

**设计要点**：

| 要点 | 说明 |
|------|------|
| 零配置 | 用户只需要输入 IP，不需要懂任何安全概念 |
| 自然语言驱动 | "帮我看看服务器安不安全" 就够了 |
| 即时反馈 | 3 分钟内出结果，不能让用户等 |
| 风险翻译 | 不展示 CVE 编号、不展示技术细节，用大白话说风险 |
| 行动指引 | 每个风险都附带"怎么办"，不需要用户自己想 |

### 1.7 合规评分模型

#### 1.7.1 评分公式

```
合规总分 = (技术域分数 x 0.5 + 管理域分数 x 0.35 + 物理域分数 x 0.15) x 100

域分数 = SUM(条款得分 x 条款权重) / SUM(条款权重)
```

#### 1.7.2 条款得分规则

| 判定结果 | 得分 | 说明 |
|----------|------|------|
| 符合（PASS） | 1.0 | 完全满足要求 |
| 部分符合（PARTIAL） | 0.5 | 部分满足，有改进空间 |
| 不符合（FAIL） | 0.0 | 完全不满足 |
| 纸上合规（PAPER_COMPLIANT） | -0.5 | 高级版专属，配置符合但防线可被绕过，扣分 |
| 未检测（NOT_TESTED） | 0.0 | 不计入总分，但拉低整体覆盖率 |

#### 1.7.3 条款权重（三级示例）

| 条款 | 权重 | 理由 |
|------|------|------|
| 8.1.4.1 身份鉴别 | 3.0 | 核心安全要求，弱口令直接导致入侵 |
| 8.1.3.1 边界防护 | 2.5 | 边界是防线第一道门 |
| 8.1.4.3 安全审计 | 2.0 | 审计是事后追溯的关键 |
| 8.1.3.3 入侵防范 | 2.0 | 入侵检测是主动防御 |
| 8.1.7.1 安全管理制度 | 1.5 | 管理域基础 |
| 8.1.4.9 数据备份恢复 | 1.5 | 业务连续性保障 |
| 其他条款 | 1.0 | 默认权重 |

#### 1.7.4 分数等级映射

| 分数区间 | 等级 | 含义 |
|----------|------|------|
| 90-100 | 优秀 | 高度合规，可应对正式测评 |
| 75-89 | 良好 | 基本合规，少量整改即可 |
| 60-74 | 一般 | 存在较多不符合项，需重点整改 |
| 40-59 | 较差 | 合规风险高，建议立即整改 |
| < 40 | 危险 | 严重不合规，可能无法通过正式测评 |

## 三、法律合规框架（新增）

### 3.1 为什么自用产品也需要法律框架

即使产品交付给用户自行使用，平台运营方仍需保护自身：

1. **连带风险**：用户使用平台对非授权目标发起渗透，平台方可能被追责
2. **采购审查**：三级等保客户在采购安全产品时，会审查产品的授权管控机制
3. **产品化授权**：将授权流程内嵌到产品中，既是法律保护，也是产品专业度的体现

### 3.2 三级授权体系设计

授权流程按操作风险等级递进，避免过度设计影响用户体验：

| 授权级别 | 风险等级 | 适用操作 | 用户流程 | 版本 |
|----------|----------|----------|----------|------|
| L1 资产所有权验证 | 低风险 | 端口扫描、配置核查、漏洞扫描 | DNS TXT / 文件验证 / 云 API 验证 | 轻量版 + 高级版 |
| L2 在线确认 | 中风险 | 弱口令检测、密码策略检查 | 勾选同意 + 短信验证码 | 轻量版 + 高级版 |
| L3 正式授权 | 高风险 | 渗透测试（PoC 利用、横向移动） | 电子授权书 + 电子签章 | 仅高级版 |

#### 3.2.1 L1 资产所有权验证（低风险）

**适用场景**：所有只读操作，不主动探测目标

| 验证方式 | 适用场景 | 实现机制 | 用户流程 |
|----------|----------|----------|----------|
| DNS TXT 验证 | 域名类资产 | 在域名 DNS 添加指定 TXT 记录，平台定期校验 | 复制 TXT 值 -> 添加 DNS 记录 -> 平台验证通过 |
| 文件验证 | Web 站点 | 在网站根目录放置平台指定的验证文件 | 下载验证文件 -> 上传到网站根目录 -> 平台访问验证 |
| 端口响应验证 | IP 资产 | 在指定端口返回平台下发的挑战字符串 | 配置服务返回指定字符串 -> 平台连接验证 |
| 云 API 验证 | 云资源 | 通过 AK/SK 调用云 API 查询资源归属（高级版） | 输入 AK/SK -> 平台查询资源列表 -> 自动关联 |

**验证通过后**：资产标记为"已验证"，可执行所有 L1 操作，无需额外授权。

#### 3.2.2 L2 在线确认（中风险）

**适用场景**：主动探测操作，可能对目标产生轻微影响

| 操作类型 | 风险说明 | 确认流程 |
|----------|----------|----------|
| 弱口令检测（hydra） | 尝试登录，可能触发账户锁定 | 勾选"我确认已获授权" + 短信验证码 |
| 密码策略检查 | 读取系统配置，无主动影响 | 勾选"我确认已获授权" |
| 漏洞利用验证（非破坏性） | 发送 PoC 请求，不执行 payload | 勾选"我确认已获授权" + 短信验证码 |

**确认记录**：系统记录确认时间、IP、操作类型，作为审计依据。

#### 3.2.3 L3 正式授权（高风险）

**适用场景**：渗透测试操作，可能对目标产生实质性影响

**授权流程**：

```
+--------------+    +--------------+    +--------------+    +--------------+
| 创建测试计划  |--->| 填写授权信息  |--->| 电子签章      |--->| 激活测试任务  |
+--------------+    +--------------+    +--------------+    +--------------+
                           |
                    +------+------+
                    |  授权书内容  |
                    +-------------+
                    | 测试范围     |
                    | 测试时间     |
                    | 禁止操作     |
                    | 免责条款     |
                    | 电子签章     |
                    +-------------+
```

**授权书核心要素**：

- **测试范围**：精确到 IP/域名/端口，不可超出授权范围
- **测试时间窗口**：精确到小时，过期自动停止
- **禁止操作清单**：如禁止 DoS、禁止修改数据、禁止触碰生产数据库
- **紧急联系人**：7x24 联系方式，出现异常立即停止
- **免责条款**：明确平台方仅提供工具，操作由用户授权并负责

### 5.2 MCP 韧性控制层（强化设计）

#### 4.2.1 健康检查与熔断

| 熔断条件 | 阈值 | 恢复策略 |
|----------|------|----------|
| 连续失败次数 | >= 3 次 | 30 秒后进入 HALF_OPEN |
| 错误率 | >= 50%（1 分钟窗口） | 60 秒后进入 HALF_OPEN |
| 响应超时 | P99 > 30s | 自动降级为缓存结果或跳过 |
| 心跳丢失 | 连续 3 次无响应 | 标记为 DOWN，告警通知 |

熔断状态：
- CLOSED（正常）：所有请求正常转发
- OPEN（熔断）：直接返回降级响应，不请求 MCP Server
- HALF_OPEN（半开）：放行 1 个探测请求，成功则恢复

#### 4.2.2 降级策略

| 场景 | 降级方案 |
|------|----------|
| 云 API 不可用 | 降级为黑盒探测 + 截图验证 |
| 渗透工具不可用 | 降级为漏洞扫描（不执行 exploit） |
| LLM 服务不可用 | 降级为纯规则引擎判定，标记"待 AI 复核" |
| OCR/VLM 不可用 | 降级为人工审核截图，标记"待人工确认" |

#### 4.2.3 异步任务队列

对于长时运行任务（渗透测试、大规模扫描），采用异步执行 + 回调通知：

```
+----------+    +--------------+    +--------------+    +----------+
| Agent    |--->| 任务队列      |--->| Worker 执行   |--->| 结果回调  |
| 提交任务  |    | Redis Stream |    | MCP Tool     |    | Agent    |
+----------+    +--------------+    +--------------+    +----------+
                       |                    |
                +------+------+      +------+------+
                | 进度上报     |      | 超时监控     |
                | WebSocket    |      | 告警通知     |
                +-------------+      +-------------+
```

| 任务类型 | 预估耗时 | 执行方式 |
|----------|----------|----------|
| 端口扫描 (nmap) | 10s ~ 5min | 同步/异步 |
| 漏洞扫描 (nuclei) | 1min ~ 30min | 异步 |
| 弱口令爆破 (hydra) | 1min ~ 1hr | 异步 + 进度上报 |
| 渗透验证 (PoC) | 5min ~ 2hr | 异步 + HITL 确认 |
| 云 API 读取 | 1s ~ 10s | 同步 |
| 截图 OCR | 3s ~ 30s | 同步 |

#### 4.2.4 结果缓存

| 缓存策略 | 缓存时长 | 适用场景 |
|----------|----------|----------|
| 云配置快照 | 1 小时 | IAM 策略、安全组规则等 |
| 扫描结果 | 24 小时 | 同一目标短期内重复扫描 |
| 等保知识库 | 7 天 | 条款解读、判定规则 |
| 漏洞库 | 1 小时 | CVE/NVD 数据 |

### 5.3 AI 大脑层：三引擎判定架构（强化设计）

#### 4.3.1 确定性判定器（规则引擎）

**职责**：处理二值逻辑、数值比较、配置校验，零幻觉。

| 检查项示例 | 规则逻辑 | 输入 | 输出 |
|------------|----------|------|------|
| 密码长度 | length >= 8 (二级) / 12 (三级) | 密码策略 API 返回 | PASS / FAIL |
| 日志留存天数 | days >= 180 | 日志配置 API 返回 | PASS / FAIL |
| 端口暴露 | port IN forbidden_ports | nmap 扫描结果 | PASS / FAIL |
| SSL 协议 | TLS_version >= 1.2 | testssl 结果 | PASS / FAIL |
| 安全组规则 | 0.0.0.0/0 NOT IN allowed_cidr | 安全组 API 返回 | PASS / FAIL |

**实现方式**：Python 规则引擎（如 `rule-engine` 库），规则以 YAML/JSON 配置，不硬编码。

```yaml
# 规则配置示例
- rule_id: "8.1.4.1_password_length"
  level: ["二级", "三级"]
  condition:
    field: "password_policy.min_length"
    operator: ">="
    value:
      二级: 8
      三级: 12
  evidence_field: "password_policy.raw_response"
  severity: "high"
```

#### 4.3.2 概率性判定器（LLM）

**职责**：处理需要语义理解、上下文推理的检查项，附带置信度分数。

| 检查项示例 | LLM 任务 | 输入 | 输出 |
|------------|----------|------|------|
| 安全管理制度是否覆盖某条款 | 语义匹配 | 制度文档 + 条款要求 | PASS/FAIL + 置信度 + 理由 |
| 网络拓扑是否满足冗余要求 | 图像理解 | 拓扑图 + 冗余要求 | PASS/FAIL + 置信度 + 标注 |
| 应急预案是否完整 | 结构化审查 | 预案文档 + 检查清单 | 缺失项列表 + 置信度 |
| 岗位职责是否分离 | 语义分析 | 岗位说明 + 三权分立要求 | PASS/FAIL + 冲突说明 |

**输出格式约束**：

```json
{
  "judgment": "FAIL",
  "confidence": 0.85,
  "reasoning": "制度文档第3.2节仅描述了系统管理员职责，未涉及安全管理员和审计管理员的职责划分，不满足三权分立要求。",
  "evidence_refs": ["doc_section_3.2", "requirement_8.1.8.2"],
  "suggestion": "建议在安全管理制度中补充安全管理员和审计管理员的职责定义，确保系统管理、安全管理、审计管理三权分立。"
}
```

**置信度阈值**：
- >= 0.9：自动判定
- 0.7 ~ 0.9：标记"AI 建议，待人工确认"
- < 0.7：标记"需人工审核"，不自动判定

#### 4.3.3 混合判定器

**职责**：规则引擎先筛，LLM 兜底，兼顾准确性和覆盖面。

```
输入数据
   |
   v
+--------------+
| 规则引擎先筛  |--- 命中确定性规则 ---> 直接输出结论（PASS/FAIL）
+------+-------+
       | 未命中
       v
+--------------+
| LLM 语义判定  |--- 置信度 >= 0.9 ---> 自动判定
+------+-------+
       | 置信度 < 0.9
       v
+--------------+
| 人工审核队列  |--- 人工判定 ---> 最终结论
+--------------+
```

### 5.5 Multi-Agent 协作设计（新增）

本节定义 L1 Agent 与 L2 Skill 之间的动态协作机制，解决"什么时候触发什么机制"的问题。

#### 4.5.1 Coordinator 模式映射

L1 Agent 采用 Coordinator 模式，角色约束如下：

| 角色 | 职责 | 禁止行为 |
|------|------|----------|
| L1 Agent（协调者） | 意图理解、任务规划、派发 Skill、接收结果、合成判定 | 禁止直接调用 MCP 工具、禁止跳过 Skill 直接出结论 |
| L2 Skill（Worker） | 执行具体检查逻辑、调用 MCP 工具、返回结构化结果 | 禁止调用其他 Skill、禁止直接与用户交互、禁止修改全局看板 |

**核心原则**：Agent 必须亲自"合成"判定结果，不能"转发" Skill 的结论。Agent 需要理解 Skill 返回的证据链，结合等保知识库做出最终判定。

#### 4.5.2 Agent 生命周期与触发机制全景


#### 4.5.3 机制A：并行 vs 串行决策

**触发时机**：阶段2，Agent 生成任务依赖图时，每次规划必触发。

| 触发条件 | 决策 | 场景举例 |
|----------|------|----------|
| 多个检查项之间**无数据依赖** | 并行派发 | 身份鉴别、访问控制、审计日志 -> 三个 Skill 同时派发 |
| 检查项之间**有数据依赖** | 串行执行 | 先 nmap 扫端口 -> 发现 22 端口 -> 再 hydra 爆破 |
| 多个**同类型资产**需要检查 | 并行派发 | 5 台 ECS 的安全配置检查 -> 5 个 Skill 实例并行 |
| **渗透验证类**任务 | 串行（需人工确认） | PoC 利用必须等 HITL 确认后才执行下一步 |

**判断逻辑**：Agent 在规划阶段为每个任务标注前置依赖，无依赖的任务在同一轮 LLM 调用中同时派发。

#### 4.5.4 机制B：Skill 工具权限过滤

**触发时机**：阶段3，Skill 实例创建时（一次性，不是每次调用）。

| 触发条件 | 过滤策略 | 场景举例 |
|----------|----------|----------|
| 所有 Skill 实例创建 | **全局黑名单**生效 | 技术域核查 Skill 不能调用"管理域模板 Skill"（禁止 Skill 互调） |
| 创建**后台异步** Skill | **异步白名单**生效 | 后台跑的 nuclei 扫描只能用只读工具，不能调用 `run_poc_exploit` |
| 用户**自定义** Skill | **加严黑名单**生效 | 客户自己写的检查规则，额外禁止访问云 API 凭证 |

**工具权限三道门**：

```
所有工具
    |
    v
+---------------------------+
| 第一道门: 全局黑名单       |
| - 禁止调用其他 Skill       |
| - 禁止直接与用户交互       |
| - 禁止修改全局看板状态     |
+---------------------------+
    | 通过
    v
+---------------------------+
| 第二道门: 自定义Skill加严  |
| - 额外禁止访问云API凭证    |
| - 额外禁止执行高危操作     |
+---------------------------+
    | 通过
    v
+---------------------------+
| 第三道门: 异步白名单       |
| - 仅允许只读/低风险工具    |
| - 默认拒绝，白名单放行    |
+---------------------------+
    | 通过
    v
最终工具池（量身定制）
```

#### 4.5.5 机制C：Skill 上下文隔离

**触发时机**：阶段3，与机制B 同时，Skill 实例初始化时。

| 状态字段 | 决策 | 场景举例 |
|----------|------|----------|
| 工具调用历史 | **克隆**（独立） | Skill A 调用 nmap 的结果不影响 Skill B 的视图 |
| 全局看板状态 | **Skill 关闭写权限** | Skill 不能直接修改合规看板，结果统一由 Agent 汇总后更新 |
| 任务队列注册 | **保留** | 异步 Skill（如渗透测试）需要注册到任务队列 |
| 实例 ID + 深度 | **独立分配** | 每个 Skill 实例有唯一 ID，深度 = 1（Agent 是 0，Skill 是 1，禁止 Skill 再派 Skill 深度=2） |

**隔离原则**：不按"整体"隔离，而是按每个状态的语义单独决策。对着父 Agent 的每项状态问："Skill 拿这个状态干啥？会不会影响父？"

#### 4.5.6 机制D：长时任务自动后台化

**触发时机**：阶段4，Skill 开始执行后，按时间阈值自动触发。

| 触发条件（预估耗时） | 行为 | 场景举例 |
|---------------------|------|----------|
| < 30s | 同步等待，直接返回结果 | 云 API 读取密码策略、安全组规则 |
| 30s ~ 2min | 前台展示进度条，Agent 等待 | nmap 扫描单台主机 |
| > 2min 且 < 30min | 自动转后台，Agent 先处理其他任务 | nuclei 漏洞扫描、nuclei 全量扫描 |
| > 30min | 直接后台 + 进度上报 | hydra 爆破、渗透验证全流程 |

**关键**：转后台后 Agent 不阻塞，可以继续给用户回复中间结果或处理其他检查项。Skill 完成后通过通知机制（机制E）回传。

#### 4.5.7 机制E：Agent-Skill 消息通信

**触发时机**：阶段5，每个 Skill 完成时自动触发。

**通信模型**：

| 方向 | 机制 | 说明 |
|------|------|------|
| Agent -> Skill（派任务） | 写入 Skill 消息队列 | 异步不阻塞，Agent 扔完走人 |
| Skill -> Agent（结果回报） | 结构化通知注入 Agent 对话流 | Skill 完成后自动触发 |

**结果通知格式**：

```json
{
  "type": "skill_completion",
  "skill_id": "skill-identity-check-001",
  "skill_type": "tech_domain_identity_auth",
  "status": "completed",
  "clauses_checked": ["8.1.4.1", "8.1.4.2"],
  "findings": [
    {"clause": "8.1.4.1", "judgment": "FAIL", "severity": "high", "evidence_ids": ["ev-001", "ev-002"]},
    {"clause": "8.1.4.2", "judgment": "PASS", "severity": "info", "evidence_ids": ["ev-003"]}
  ],
  "usage": {"tool_calls": 5, "duration_ms": 45000},
  "needs_follow_up": true,
  "follow_up_suggestion": "发现弱口令，建议触发深度验证 Skill 进行渗透确认"
}
```

Agent 收到后：
1. 解析 findings，更新内部状态
2. 检查 `needs_follow_up`，决定是否触发机制F
3. 检查是否所有任务完成，决定是否进入阶段7合成

#### 4.5.8 机制F：Skill 实例复用决策（Continue vs Spawn）

**触发时机**：阶段6，Agent 收到 Skill 结果后，发现需要追加检查时触发。

| 触发条件 | 决策 | 场景举例 |
|----------|------|----------|
| 新任务与当前 Skill **同一资产 + 同一方向** | **复用**（发消息续命） | 身份鉴别 Skill 发现弱口令 -> 追加"检查是否启用 MFA" |
| 新任务与当前 Skill **同一资产 + 不同方向** | **新建** | 身份鉴别 Skill 完成 -> 需要对该资产做访问控制检查 -> 新建 Skill |
| 新任务是**验证类**（需要"新鲜视角"） | **新建** | 配置检查 Skill 完成 -> 需要渗透验证 Skill 交叉确认 -> 必须新建 |
| 当前 Skill 上下文已经**很大**（接近 token 上限） | **新建** | Skill 已经检查了 20 个条款，上下文快满了 -> 新建 Skill 处理剩余条款 |

#### 4.5.9 Prompt 缓存友好设计

**触发时机**：每次 LLM 调用时，架构层面保证。

| 设计原则 | 实现方式 |
|----------|----------|
| 不变内容前置 | 等保知识库、判定规则、工具 Schema 放在 system prompt 前部 |
| 动态内容后置 | 项目信息、资产列表、当前任务放在 system prompt 后部 |
| 同一租户复用 | 同一租户的多次调用尽量保持 prompt 前缀字节一致，复用 LLM API 缓存 |

**成本收益**：缓存命中时，API 成本降至 10%，延迟大幅降低。

#### 4.5.10 触发条件全景表

| 生命周期阶段 | 触发机制 | 触发条件 | 判断输入 |
|-------------|----------|----------|----------|
| 任务规划 | 并行/串行决策 | 每次规划必触发 | 任务依赖图（哪些任务有前置依赖） |
| Skill 创建 | 工具权限过滤 | 每次创建必触发 | Skill 类型（内置/自定义/异步） |
| Skill 创建 | 上下文隔离 | 每次创建必触发 | 父 Agent 的运行时状态 |
| Skill 执行 | 自动后台化 | 执行时间超过阈值 | 预估耗时 vs 实际耗时 |
| Skill 完成 | 结果通知 | 每次完成必触发 | Skill 执行结果 |
| 结果回收 | 复用/新建决策 | 需要追加检查时触发 | 任务相关性 + 方向 + 上下文大小 |
| 全部完成 | 合成判定 | 所有 Skill 完成后必触发 | 所有 findings + 证据链 |

**核心原则**：并行决策和权限过滤是"每次必触发"的硬规则，自动后台化和复用决策是"条件触发"的软规则。

#### 5.6.2 编排模式

### 6.1 运行时序

### 6.2 鉴权校验清单

每次 MCP 工具调用前，MCP Gateway 执行以下校验：

| 校验项 | 校验逻辑 | 失败处理 |
|--------|----------|----------|

| 频率限制 | 同一工具调用频率未超限 | 拒绝调用，提示稍后重试 |

## 九、整改闭环设计（新增）

### 9.1 整改闭环流程

#### 9.1.1 角色与职责

| 角色 | 整改职责 |
|------|----------|
| Admin | 分配责任人、查看全局进度、导出报告 |
| Operator | 执行整改、反馈结果、申请复测 |
| Approver | 审批整改方案、确认整改完成、授权复测 |
| Viewer | 查看整改进度、查看报告（只读） |

#### 9.1.2 整改流程

```
发现不符合项
    |
    v
+------------------+
| Admin 分配责任人  |  指定 Operator，设置整改期限
+------------------+
    |
    v
+------------------+
| Operator 查看方案 |  查看 AI 生成的整改方案，确认理解
+------------------+
    |
    v
+------------------+
| Operator 执行整改 |  按方案执行，记录整改过程
+------------------+
    |
    v
+------------------+
| Operator 标记完成 |  上传整改证据（截图/配置/日志）
+------------------+
    |
    v
+------------------+
| Approver 审批     |  审核整改证据，确认是否有效
+------------------+
    |
    +-- 审批通过 --> 触发复测 --> 复测通过 --> 关闭工单
    |
    +-- 审批退回 --> Operator 重新整改
```

#### 9.1.3 通知机制

| 事件 | 通知对象 | 通知方式 | 通知内容 |
|------|----------|----------|----------|
| 新发现高危不符合项 | Admin | 钉钉/飞书/邮件 | 条款、风险等级、建议责任人 |
| 工单分配 | Operator | 钉钉/飞书/邮件 | 不符合项详情、整改期限 |
| 整改完成待审批 | Approver | 钉钉/飞书/邮件 | 整改证据、复测申请 |
| 审批退回 | Operator | 钉钉/飞书/邮件 | 退回原因、改进建议 |
| 复测通过 | Admin + Operator | 钉钉/飞书/邮件 | 合规分数更新、工单关闭 |
| 整改超期未完成 | Admin | 钉钉/飞书/邮件 | 超期工单列表、风险提醒 |

## 十一、安全风控与 AI 可靠性设计（强化版）

### 11.2 AI 可靠性保障

| 保障机制 | 实现方式 | 说明 |
|----------|----------|------|
| 证据链强绑定 | 每个判定必须附带原始证据（API 响应/日志/截图） | 无证据不判定 |

### 11.3 代码安全规范（新增）

作为渗透测试平台，代码安全尤为重要——如果平台自身有漏洞，会被"以子之矛攻子之盾"。

#### 11.3.1 输入验证与输出编码

| 风险类型 | 防护措施 | 实现方式 |
|----------|----------|----------|
| SQL 注入 | 参数化查询，禁止字符串拼接 | SQLAlchemy ORM + 参数绑定 |
| XSS 攻击 | 输出编码，CSP 头 | React 自动转义 + Content-Security-Policy |
| CSRF 攻击 | Token 验证 | JWT Bearer Token + SameSite Cookie |
| 命令注入 | 参数白名单，禁止 shell 拼接 | MCP 工具参数校验 + subprocess 参数化 |
| 路径遍历 | 路径规范化，白名单目录 | os.path.realpath + 目录白名单校验 |
| 文件上传 | 类型/大小限制，病毒扫描 | MIME 校验 + ClamAV 扫描 |

#### 11.3.2 认证与授权安全

| 安全要求 | 实现方式 |
|----------|----------|
| 密码存储 | bcrypt 哈希，不可逆 |
| 密码强度 | ≥12位，含大小写+数字+特殊字符 |
| 会话管理 | JWT 短期 Access Token + 长期 Refresh Token |
| 权限控制 | RBAC 四级角色（Admin/Operator/Approver/Viewer） |
| 多因素认证 | 高级版强制 MFA（TOTP） |
| 登录保护 | 5次失败锁定15分钟 + IP 限流 |

#### 11.3.3 敏感数据处理

| 数据类型 | 处理规则 |
|----------|----------|
| 用户密码 | bcrypt 哈希存储，明文不落盘 |
| 云凭证 (AK/SK) | KMS 加密存储，内存中解密使用 |
| 扫描结果 | 租户隔离，加密存储 |
| 日志数据 | 脱敏处理，禁止记录密码/凭证 |
| 测试数据 | 禁止使用生产数据，使用 Mock 数据 |

#### 11.3.4 依赖安全

| 措施 | 工具 | 频率 |
|------|------|------|
| 依赖漏洞扫描 | npm audit / pip-audit / safety | 每次 CI/CD |
| 许可证合规 | license-checker | 每次新增依赖 |
| 依赖锁定 | package-lock.json / requirements.txt | 强制 |
| 最小依赖原则 | 定期审查，移除未使用依赖 | 每月 |

#### 11.3.5 代码审查安全清单

| 检查项 | 说明 |
|--------|------|
| 硬编码凭证 | 禁止在代码中硬编码密钥/密码 |
| 不安全的反序列化 | 禁止使用 pickle/yaml.load 处理不可信数据 |
| 不安全的随机数 | 安全场景使用 secrets 模块，非 random |
| 错误信息泄露 | 生产环境禁止返回堆栈信息 |
| 日志注入 | 日志内容转义，禁止用户输入直接写入 |

#### 11.3.6 安全测试

| 测试类型 | 工具 | 阶段 |
|----------|------|------|
| SAST（静态分析） | Bandit (Python) / ESLint Security | 每次提交 |
| DAST（动态测试） | OWASP ZAP | 每次部署 |
| 渗透测试 | 自身平台 + 第三方 | 每季度 |
| 依赖扫描 | Snyk / Dependabot | 持续 |

#### 11.3.7 密钥管理

| 场景 | 方案 |
|------|------|
| 开发环境 | .env 文件，加入 .gitignore |
| 测试环境 | 环境变量 + Vault |

## 十三、市场差异化与护城河

| 竞品类型 | 典型代表 | 局限性 | 我们的突破点 |
|----------|----------|--------|--------------|
| CSPM | 阿里云CSPM、深信服 | 只做白盒配置读取，得出"配置符合"假象 | 深度验证 Skill，打出"纸上合规"原形 |
| BAS | SafeBreach、腾讯云RAS | 离合规太远，是安全团队的攻防玩具 | 攻防结果自动映射回等保条款 |
| AI助手 | 超测智能、天帷智审 | 只解决最后1公里报告撰写 | 解决前99公里证据采集与验证 |

### 13.1 护城河构建

| 阶段 | 护城河 | 说明 |
|------|--------|------|
| 短期 | 工具链整合 | MCP 架构快速接入新工具，覆盖面持续扩大 |
| 中期 | 知识库积累 | 等保条款解读、判定规则、整改方案库持续沉淀 |
| 长期 | 数据飞轮 | 客户使用数据 -> 优化判定模型 -> 更准确的合规评估 -> 更多客户 |

### 13.2 增长漏斗（AARRR）

| 阶段 | 目标 | 关键指标 | 设计要点 |
|------|------|----------|----------|
| Acquisition（获客） | 获取流量 | 注册数 | 免费扫描钩子、云市场上架、测评机构推荐 |
| Activation（激活） | 到达 Aha Moment | 首次扫描完成率 | Onboarding 引导、3 分钟出结果 |
| Retention（留存） | 持续使用 | 月活、复测率 | 持续监控、到期预警、整改跟踪 |
| Revenue（变现） | 付费转化 | 付费率、ARPU | 免费 -> 轻量版 -> 高级版升级路径 |
| Referral（推荐） | 口碑传播 | NPS、推荐数 | 测评机构合作、报告分享功能 |

### B.1 核心数据模型

```
Tenant (租户)
+-- id
+-- name
+-- subscription_tier (light/advanced)
+-- cloud_credentials (encrypted)
+-- created_at

Project (项目)
+-- id
+-- tenant_id
+-- name
+-- compliance_level (二级/三级)
+-- status (active/archived)
+-- created_at

Asset (资产)
+-- id
+-- project_id
+-- type (ip/domain/cloud_resource)
+-- value
+-- ownership_verified (bool)
+-- verification_method
+-- verified_at

Authorization (授权书)
+-- id
+-- project_id
+-- scope (JSON: 目标范围)
+-- time_window_start
+-- time_window_end
+-- signed_at
+-- signature_cert
+-- status (active/expired/revoked)

ScanTask (扫描任务)
+-- id
+-- project_id
+-- type (full/incremental/targeted)
+-- status (pending/running/completed/failed)
+-- started_at
+-- completed_at
+-- triggered_by (manual/scheduled/event)

Finding (发现项)
+-- id
+-- project_id
+-- clause_id (等保条款编号)
+-- severity (critical/high/medium/low)
+-- judgment (pass/fail/partial/not_tested/paper_compliant)
+-- judgment_engine (rule/llm/hybrid/manual)
+-- confidence (float, for LLM judgments)
+-- evidence_ids (JSON array)
+-- description
+-- remediation_suggestion
+-- status (open/in_progress/resolved/false_positive)
+-- created_at

Evidence (证据)
+-- id
+-- finding_id
+-- type (tool_output/screenshot/api_response/log)
+-- source (tool_name or manual)
+-- content (JSON or file reference)
+-- hash (SHA-256, for integrity)
+-- created_at

RemediationTicket (整改工单)
+-- id
+-- finding_id
+-- assignee
+-- status (pending/in_progress/resolved/verified)
+-- external_ticket_id (Jira/飞书/钉钉)
+-- ai_remediation_plan (JSON)
+-- resolved_at
+-- verified_at

ComplianceSnapshot (合规快照)
+-- id
+-- project_id
+-- snapshot_time
+-- total_score
+-- clause_results (JSON)
+-- triggered_by (manual/scheduled/event)
```
